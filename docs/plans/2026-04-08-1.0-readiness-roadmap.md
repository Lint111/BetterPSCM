# BetterPSCM 1.0.0 Readiness Roadmap

**Date:** 2026-04-08
**Starting point:** v0.4.0 (`b82c18b`), the robustness-roadmap release
**Target:** v1.0.0 — marketplace-stable, `preview: false`

## Current state (0.7 of the way to 1.0)

v0.4.0 shipped the structural foundation a 1.0 requires: a clean
backend abstraction, `PlasticContext` replacing module-level cm globals,
a shared destructive-ops safety layer with backup/audit/bulk-guard,
integration tests driving a real cm binary, and the 24+ follow-through
fixes from the March 20 audit.

What's missing is the **operational maturity** a 1.0 implies — CI,
security headers, coverage of the remaining singletons, marketplace
validation, and documentation for end users and contributors.

### Grade by dimension

| Dimension | Grade | Evidence |
|---|---|---|
| Core engineering | 1.0 | 465 unit + 14 integration tests, clean layering, ctx-injectable backend, audit discipline |
| Polish | 0.8 | Known structural gaps listed below, all bounded and fixable |
| Operational | 0.5 | No CI, no marketplace publish history, no telemetry, no contribution guide |

## Burndown plan

Three focused working days close the gap honestly. Items are grouped
by priority. Each item carries a problem statement, a proposed
approach, and verification criteria so a future session (or a different
contributor) can pick up any single item and execute it without
reconstructing context.

### Day 1 — Mandatory hygiene (~3–4 hours)

#### 1. Add Content-Security-Policy to BetterPanel base class

**Problem:** `grep -c 'Content-Security-Policy\|nonce\|cspSource'` across
`src/views/` returns zero. Every webview runs with `enableScripts: true`
and no CSP. VS Code doesn't require CSP but the marketplace publishing
guidelines flag its absence as a quality issue. More importantly, if a
webview ever interpolates user-controlled text into HTML without
escaping, there's no defense-in-depth layer.

**Approach:** Inject a default CSP into the HTML `<head>` from
`BetterPanel` before the subclass HTML is written. Generate a per-panel
nonce and expose it as `this.nonce` for subclasses to interpolate into
their `<script nonce="...">` tags. The default policy should allow:

- `default-src 'none'`
- `img-src ${webview.cspSource} https: data:`
- `style-src ${webview.cspSource} 'unsafe-inline'` (VS Code webview theme vars need inline styles)
- `script-src 'nonce-<generated>'`
- `font-src ${webview.cspSource}`

Subclasses keep their inline `<script>` blocks but the base class
rewrites them to carry the nonce. The `historyGraphPanel` uses a
`WebviewView` not `WebviewPanel`, so it needs a parallel helper rather
than extending BetterPanel.

**Verification:**
- Existing webview tests still pass
- A new test inspects the rendered HTML and asserts a `Content-Security-Policy` `<meta>` tag is present with a unique `nonce-*` per panel instance
- Manual smoke test: open CodeReviewPanel and ReviewSnippetPanel in a real VS Code instance, confirm scripts still run

**Files:** `src/views/panels/betterPanel.ts`, `src/views/codeReviewPanel.ts`,
`src/views/reviewSnippetPanel.ts`, `src/views/historyGraphPanel.ts`,
`test/unit/views/betterPanel.test.ts`

#### 2. GitHub Actions CI

**Problem:** `.github/workflows/` does not exist. Every test run happens
locally; there is no automated gate for PRs or pushes to `main`. A 1.0
release without CI is a red flag for any reviewer.

**Approach:** Add `.github/workflows/ci.yml` with two jobs:

1. **`unit`** — runs on push and PR. `npm ci`, `npx tsc --noEmit`, `npm test`. Matrix over Node 18 + 20 (match the `engines` field).
2. **`build`** — runs on push and PR. `npm ci`, `npm run build:ext`, upload `dist/` as an artifact.

Integration tests deliberately stay OFF CI for now because they need a
real `cm` binary and a Plastic workspace, neither of which is available
on GitHub's runners. Add them later when we have a mock server or a
self-hosted runner.

**Verification:**
- PR opens and both jobs run green
- Deliberately break a test on a branch → CI fails → merge is blocked

**Files:** `.github/workflows/ci.yml`

#### 3. Decide: single-root-only vs multi-root workspace support

**Problem:** `src/extension.ts` uses `vscode.workspace.workspaceFolders?.[0]`
in three places. If a user opens a multi-root workspace, the extension
silently binds to whichever folder happens to be first. No error, no
warning, just confusing behavior.

**Approach (option A — document the limitation, ~30 min):**

- Add a `BetterPSCM.activate: unsupported multi-root workspace`
  warning if `workspaceFolders.length > 1`
- Update README with "multi-root workspaces are not supported yet"
- Add `"workspaceContains:**/.plastic/plastic.workspace"` to `activationEvents` (maybe already there — verify)

**Approach (option B — actually support multi-root, ~1 day):**

- Iterate every workspace folder on activate, detect `.plastic/`
- Build one `PlasticContext` per folder
- Keep a `Map<folderUri, { backend, provider, context }>` keyed by folder URI
- Register multiple `SourceControl` instances, one per folder
- All commands resolve the right context from `resource.uri` before dispatching

Option A is the sane 1.0 answer. Option B is correct but is a feature,
not a polish pass — defer to a later release.

**Verification:**
- Open a single-root Plastic workspace → extension activates normally
- Open a multi-root workspace with one Plastic folder + one non-Plastic folder → extension shows the warning and activates against the Plastic folder
- Open two Plastic folders in one multi-root workspace → extension shows the warning and activates against the first

**Files:** `src/extension.ts`, `README.md`

#### 4. Marketplace publish dry-run

**Problem:** The extension has never been installed via
`code --install-extension betterpscm-<version>.vsix` on a clean VS Code
instance. All testing happens in the F5 Extension Development Host,
which has a different lifecycle and doesn't catch bundling issues,
missing files, or packaging metadata errors.

**Approach:**

1. `npm run package` → produces `betterpscm-0.4.0.vsix`
2. Spin up a clean VS Code profile: `code --profile bpscm-test-profile`
3. Install from the `.vsix`
4. Open a Plastic workspace
5. Walk through the full feature matrix: stage, checkin, branch switch,
   history graph, code reviews, clean stale, MCP server
6. Check the VS Code developer tools console for errors
7. Uninstall, reinstall to catch first-run state issues

Document any findings and fix them before the 1.0 publish.

**Verification:**
- Installation succeeds without warnings in the VS Code install log
- Every feature works in the clean profile
- No extension host console errors during normal use

**Files:** none (this is a verification step)

### Day 2 — Architectural debt close-out (~5–6 hours)

#### 5. Gap 4 phase 3: migrate auth/client/config to PlasticContext

**Problem:** `src/api/client.ts` has 5 module-level `let`s, `src/api/auth.ts`
has 4, `src/util/config.ts` has 2. That's **11 module-level singletons**
that Gap 4 phase 2 did not touch. Today they don't break anything
because the extension and MCP server each run in their own process.
Tomorrow they:

- Block any form of multi-workspace support
- Leak state between unit tests that don't remember to reset every module
- Make the auth state machine harder to test because mocking requires
  module-level stubs

**Approach:** Extend `PlasticContext` to carry the auth + client + config
state:

```typescript
export interface PlasticContext {
  readonly workspaceRoot: string;
  readonly cmPath: string;
  readonly auth: AuthState;         // new
  readonly apiClient: PlasticClient; // new
  readonly config: WorkspaceConfig;  // new
}
```

Then migrate one module at a time, leaves first:

1. `util/config.ts` — `initDetectedConfig` and `getConfig` become methods on a `WorkspaceConfig` class. Module globals removed.
2. `api/client.ts` — `getClient` / `setOrgNameHints` / `setResolvedOrgName` become methods on a `PlasticClient` class. `clientInstance` module global removed.
3. `api/auth.ts` — `hasStoredCredentials` / `loginWithPAT` / `logout` become methods on an `AuthState` class that owns `cachedAccessToken` / `cachedRefreshToken` / `refreshPromise`. `initAuth(secretStorage)` becomes the constructor.

`extension.ts` builds one of each during `activate()` and bundles them
into the `PlasticContext`. `mcp/server.ts` does the same in `main()`.

This is the biggest single item on the roadmap. Do it as a standalone
commit with its own audit pass.

**Verification:**
- `grep -c "^let " src/api/ src/util/config.ts` returns 0
- Existing unit tests still pass without changes (all 465)
- Integration tests still pass
- A new test constructs two independent contexts with different auth
  states and verifies they don't leak into each other

**Files:** `src/core/context.ts`, `src/api/client.ts`, `src/api/auth.ts`,
`src/util/config.ts`, `src/extension.ts`, `src/mcp/server.ts`, plus
any test mocks that touch module state

**Effort:** ~4 hours of focused refactor + audit

#### 6. Migrate `branchSwitch "Discard"` to the shared destructive-ops layer

**Problem:** `branchSwitch.ts` has three flows: "Check In & Switch",
"Shelve & Switch", "Discard & Switch". The Discard flow calls
`undoCheckout(realChangePaths)` directly with inline confirmation logic
and no backup. Meanwhile the UI's Clean Stale command and the MCP's
`bpscm_clean_stale` both go through `executeDestructiveRevert` which
provides backup, audit, bulk guard, and Unity classification. This is
the last asymmetry in the destructive-ops posture.

**Approach:** Replace the direct `undoCheckout` call in the Discard path
with `executeDestructiveRevert`:

```typescript
const revertResult = await executeDestructiveRevert({
  tool: 'discard_on_branch_switch',
  files: realChangePaths,
  backend: getBackend(),
  workspaceRoot: wsFolder.uri.fsPath,
  workspaceName: branchName,
  audit: extensionAuditLogger,
});
```

The shelve path is a separate question — shelving already persists the
content via `cm shelveset save`, so adding a backup layer on top is
redundant. Leave the shelve path alone.

**Verification:**
- Click "Discard & Switch" during a branch switch, confirm the Unity
  warning and bulk guard appear in the confirmation dialog
- Confirm a backup is written to `~/.plastic-scm-backups/`
- Confirm the audit log entries appear in the Plastic SCM output channel
- Existing branch switch tests still pass

**Files:** `src/commands/branchSwitch.ts`

#### 7. Validate or remove the `HD` change type mapping

**Problem:** `src/core/backendCli.ts:46` maps `'HD': 'changed'` in
`CM_CHANGE_TYPE_MAP`. No comment explaining what HD stands for in cm
status output. Pre-existing; not introduced this session. Might be
"HEAD" or "held" or something version-specific. If it's dead, remove
it. If it's real, document it.

**Approach:** Grep for `HD` in Plastic documentation and historical cm
output samples. If you can't find it, create a test workspace, exercise
every cm command that produces status output, and see if HD ever
appears. If it never does, delete the map entry.

**Verification:**
- Either a comment on the map entry explaining what HD means, or the
  entry is gone
- If gone, existing tests still pass (no test currently relies on HD)

**Files:** `src/core/backendCli.ts`

### Day 3 — Coverage expansion (~4–5 hours)

#### 8. Integration tests for branch operations

**Problem:** The current integration tier covers undoCheckout,
destructiveOps backup, addToSourceControl, getStatus path round-trip,
and checkin lifecycle. Branch operations (`createBranch`, `switchBranch`,
`deleteBranch`, `listBranches`) have zero integration coverage. Same
for merge operations and code reviews.

**Approach:** Add `test/integration/branch.test.ts` with:

- `createBranch` → creates a branch off the current one, confirms it
  shows up in `listBranches`
- `switchBranch` → switches to the new branch, confirms
  `getCurrentBranch` reflects the change
- `deleteBranch` → deletes the branch, confirms it no longer appears
- Round-trip: createBranch → switchBranch → switchBranch back →
  deleteBranch, asserting state at every step

Use the existing fixture scaffold. The branch lifecycle doesn't create
committed files so cleanup is simple — just delete the test branches
in `afterEach`.

Merges and code reviews are more involved (need multiple branches or a
running code review) so defer to a later expansion.

**Verification:**
- `npm run test:integration` → existing 14 + new ~6 branch tests all pass

**Files:** `test/integration/branch.test.ts`

#### 9. End-to-end test of the auth state machine

**Problem:** `tryAutoLoginFromDesktopClient` has four possible outcomes
(PAT works, exchange works, both fail, no cached token). CRIT-4 from
the March 20 audit added tests for the happy path, but the full state
machine across all four outcomes isn't soak-tested. Auth flows are the
single most complex piece of state in the extension and the most likely
source of "works for me" bugs.

**Approach:** Add `test/unit/extension/authStateMachine.test.ts` with
one test per outcome. Use mocked `detectCachedToken`, mocked REST
client, and mocked `loginWithPAT`/`loginWithToken`. Assert:

- Correct path taken
- Correct log output for the user
- Correct final state (`hasCreds`, `resolvedOrgName`)
- No leaked credentials on the failure paths

This test doesn't need integration infrastructure because the whole
thing runs through the module-level auth state (soon PlasticContext).

**Verification:**
- All 4 outcomes covered with explicit assertions
- `npm test` still passes

**Files:** `test/unit/extension/authStateMachine.test.ts`

## Deferred — NOT 1.0 blockers

These are legitimate improvements but don't belong on the 1.0 critical
path. They're noted here so they don't get lost.

- **Localization scaffolding** — `package.nls.json` + runtime locale
  loading. Lower priority; English is fine for 1.0.
- **Telemetry / error reporting** — once published, you have no signal
  when users hit crashes. Opt-in only per VS Code guidelines.
- **Performance benchmarks + SLO targets** — no current evidence the
  extension is slow; measure if complaints arrive.
- **Multi-root workspace support** — option B above. Feature, not
  polish. Defer to v1.1 or later.
- **Contributing guide / code style doc** — nice to have if external
  contributors appear; premature until then.
- **`cm undocheckout` output parsing** — the optimistic-return behavior
  noted in MIN-7 of `audit-2026-04-07-07-55-betterpscm-prepush.md`. The
  survivors verification in `cleanStale` compensates. Only revisit if
  real users hit misleading post-revert UI.

## What NOT to do before 1.0

Avoid scope creep. Specifically:

- **Don't add new features.** v0.4.0 is feature-complete for the stated
  product scope. Every new feature is a new surface to test, break, and
  maintain. Ship stability first.
- **Don't refactor beyond this roadmap.** The temptation to keep
  improving the codebase is real. The goal of 1.0 is to commit to the
  current surface and stop churning it.
- **Don't skip Day 1.** Items 1–4 are the difference between "works on
  the developer's machine" and "shippable to strangers". No CI + no CSP
  + no multi-root story + no install verification is a quality bar a
  1.0 should not cross.
